## 1. UI 탐지 모듈 전환 — OmniParser YOLOv8 (Step 5-1)

### 배경

기존 Phase 1은 Claude Vision API를 사용해서 "이 이미지에서 UI 요소 유형을 알려줘"라는 방식으로 탐지를 수행하고 있었다. 이 방식의 문제점은 **탐지 단계에서부터 API를 호출**해야 한다는 것이었다. 사용자가 아이콘을 탭할 때마다 Claude API가 먼저 탐지용으로 한 번, 설명 생성용으로 한 번 호출되어 비용이 두 배로 발생했다.

Microsoft가 공개한 **OmniParser v2.0** 모델로 전환하여 탐지 단계에서 API 호출을 완전히 제거했다.

### OmniParser란

67,000장의 모바일 스크린샷으로 파인튜닝된 YOLOv8 모델이다. nc=1(단일 클래스)로 설계되어 화면의 모든 인터랙티브 요소를 "icon" 하나의 클래스로 탐지한다. 로컬에서 실행되므로 API 비용이 없다.

```
가중치 파일: backend/weights/icon_detect/model.pt
모델 출처: microsoft/OmniParser-v2.0 (HuggingFace)
```

### element_type 분류 방법

OmniParser는 단일 클래스 탐지만 하므로, 탐지된 박스의 **종횡비(가로/세로 비율)**로 element_type을 추론한다.

```
박스 종횡비 = 너비 / 높이

  비율 < 1.5  →  icon    (정사각형에 가까운 심볼. 예: 홈 아이콘, 검색 아이콘)
  비율 1.5~4  →  button  (가로로 넓은 영역. 예: 검색창, 탭 버튼)
  비율 ≥ 4    →  text    (매우 납작한 형태. 예: 텍스트 레이블)
  탐지 없음   →  unknown
```

### 반환 구조

```python
{
  "is_ui_element": True,       # UI 요소 탐지 여부
  "element_type": "icon",      # "icon" | "button" | "text" | "unknown"
  "confidence": 0.72,          # YOLOv8 탐지 신뢰도 (0.0~1.0)
  "description_hint": "심볼/그림 형태의 아이콘 요소"  # Deep Track 프롬프트 보조
}
```

### description_hint 활용

탐지된 element_type에 따라 description_hint가 자동 생성되어, 이후 Deep Track(Claude Vision) 프롬프트에 삽입된다.

```python
_ELEMENT_HINTS = {
    "icon":    "심볼/그림 형태의 아이콘 요소",
    "button":  "탭 가능한 버튼 요소",
    "text":    "텍스트 레이블",
    "unknown": "",
}
```

| 파일 | 변경 내용 |

## 2. 핵심 버그 수정 — 크롭 패딩 방식 도입

### 문제 발견

OmniParser로 전환 직후 실제 테스트에서 **탐지 신뢰도가 0.03~0.05 수준**으로 사실상 탐지에 실패하는 문제가 발견됐다. 잘 탐지될 때는 0.6 이상이 나와야 정상이었다.

### 원인 분석

OmniParser icon_detect 모델은 **전체 스크린샷을 입력으로 받도록 학습**되었다. 즉, 화면 전체에서 아이콘들의 위치를 한꺼번에 탐지하는 방식이다. 그런데 bridgeUI는 사용자가 선택한 **크롭 이미지만** 서버로 전송하기 때문에, 모델이 "전체 스크린샷"이라는 컨텍스트 없이 작은 조각 이미지 하나를 받는 상황이 되었다.

| 입력 방식 | confidence |
|-----------|-----------|
| 크롭 이미지 단독 입력 | 0.03~0.05 (탐지 실패) |
| 640×960 캔버스에 크롭을 중앙 배치 후 탐지 | 0.6+ (탐지 성공) |

### 해결 방법 — 캔버스 패딩

크롭 이미지를 640×960 크기의 회색 배경(#f0f0f0) 캔버스 중앙에 배치한 뒤 YOLOv8에 입력한다. 모델 입장에서는 "스크린샷처럼 생긴 입력"을 받게 된다. 탐지 결과에서 크롭이 배치된 영역과 겹치는 박스만 필터링하여 올바른 결과를 추출한다.

```
[원본 크롭 이미지]  →  [640×960 캔버스 중앙 배치]  →  [YOLOv8 탐지]  →  [크롭 영역 박스 필터링]
```

```python
def _pad_to_canvas(image):
    canvas = Image.new("RGB", (640, 960), "#f0f0f0")
    paste_x = (640 - image.width) // 2
    paste_y = (960 - image.height) // 2
    canvas.paste(image, (paste_x, paste_y))
    return canvas, paste_x, paste_y
```

탐지 후 크롭 영역 박스 필터링:

```python
# 크롭이 배치된 영역 (canvas 좌표 기준)
crop_x1, crop_y1 = offset_x, offset_y
crop_x2, crop_y2 = offset_x + crop.width, offset_y + crop.height

# 이 영역과 겹치는 박스 중 confidence 최고값 선택
for box in all_boxes:
    if box.overlaps(crop_region) and box.conf > best_conf:
        best_box = box
```

### 추가된 진단 스크립트

`backend/tests/test_detector.py` — 단독 입력 vs 패딩 방식 confidence 비교 진단 도구

```

```

## 3. 앱 context 기반 하이브리드 추론 파이프라인 (Step 7~8)

### 배경

가장 핵심적인 설계 문제: **같은 아이콘이 앱마다 전혀 다른 의미**를 가진다.

```
화살표 아이콘  →  네이버 지도: "우회전 안내"
화살표 아이콘  →  카카오톡: "메시지 전달(공유)"
```

이를 해결하려면 "지금 사용자가 어떤 앱을 쓰고 있는가"라는 정보가 크롭 이미지와 함께 백엔드로 전달되어야 한다.

### Android — 포그라운드 앱 감지

Android `UsageStatsManager` API를 사용해서 bridgeUI 버튼을 탭하는 시점의 **직전 실행 앱 패키지명**을 감지한다. 이 정보는 MethodChannel을 통해 Flutter로 전달된다.

#### MainActivity.kt에 추가된 핵심 코드

```kotlin
// MethodChannel에 "getForegroundApp" 핸들러 등록
"getForegroundApp" -> result.success(getForegroundApp())

private fun getForegroundApp(): Map<String, String> {
    val usm = getSystemService(USAGE_STATS_SERVICE) as UsageStatsManager
    val now = System.currentTimeMillis()
    val stats = usm.queryUsageStats(INTERVAL_DAILY, now - 60_000L, now)
    
    // bridgeUI 자기 자신을 제외하고 가장 최근 사용된 앱 선택
    val prev = stats
        ?.filter { it.packageName != packageName && it.lastTimeUsed > 0 }
        ?.maxByOrNull { it.lastTimeUsed }
    
    return mapOf(
        "package" to prev.packageName,  // "com.nhn.android.nmap"
        "name"    to appName,           // "네이버 지도"
    )
}
```

#### 필요 권한

`AndroidManifest.xml`에 추가:
```xml
<uses-permission android:name="android.permission.PACKAGE_USAGE_STATS"
    tools:node="replace" />
```

단, 이 권한은 설치 시 자동 허용되지 않는다. 사용자가 **설정 → 앱 → 특별한 앱 접근 권한 → 사용 정보 접근**에서 직접 허용해야 한다. 권한이 없으면 빈 문자열이 전달되고, 파이프라인은 앱 필터 없이 전체 검색으로 폴백한다.

### Flutter — 앱 context 수집 흐름

```
트리거 버튼 탭
    ↓
getForegroundApp() 호출 (MethodChannel)
    ↓
{ package: "com.nhn.android.nmap", name: "네이버 지도" }
    ↓
requestCapture() → FreezeOverlay(appPackage, appName)
    ↓
ExtractedElement { croppedImageBytes, metadata, appPackage, appName }
    ↓
POST /capture { file: 크롭PNG, app_package: "com.nhn.android.nmap", app_name: "네이버 지도" }
```

### Backend — Fast Track (pipeline/fast_track.py)

CLIP 벡터 유사도 검색에 `app_package` 필터를 적용한다. 동일한 아이콘이라도 **다른 앱**의 데이터는 검색 대상에서 제외된다.

```python
SIMILARITY_THRESHOLD = 0.90  # 코사인 유사도 임계값

def search(embedding, app_package=""):
    where = {"app_package": app_package} if app_package else None
    
    results = collection.query(
        query_embeddings=[embedding],
        n_results=1,
        where=where,           # ← 앱 필터 핵심
        include=["metadatas", "distances"],
    )
    
    # ChromaDB cosine 공간: distance 0 = 동일, 1 = 완전 다름
    similarity = 1.0 - distance
    
    if similarity >= 0.90:
        return { "description": ..., "similarity": ..., "element_type": ... }
    return None  # 캐시 미스
```

### Backend — Deep Track (pipeline/deep_track.py)

캐시 미스 시 Claude Vision을 호출해서 설명을 생성한다. 이때 앱 이름과 element_type을 프롬프트에 포함시켜 **앱 맥락에 맞는 설명**을 유도한다.

```python
# 앱 패키지명 → 한국어 앱 이름 매핑 테이블
_APP_NAMES = {
    "com.nhn.android.nmap": "네이버 지도",
    "com.baemin.android":   "배달의민족",
    "mobi.korail.Talk":     "코레일",
    "com.kakao.talk":       "카카오톡",
    # ... 16개 앱 매핑
}

# 생성되는 프롬프트 예시
"이것은 네이버 지도 앱의 icon입니다.
 70대 사용자가 이것을 눌렀을 때 어떤 일이 일어나는지,
 쉬운 단어로 2문장 이내로 설명하세요."
```

생성된 설명은 ChromaDB에 저장되어, 같은 앱에서 동일/유사한 이미지가 다시 들어오면 Claude 호출 없이 Fast Track으로 응답한다.

### /capture 엔드포인트 응답 구조

```json
{
  "track":        "fast",
  "description":  "원하는 목적지까지 길 안내를 받을 수 있어요. 누르면 출발지와 목적지를 입력하는 화면이 나와요.",
  "element_type": "button",
  "confidence":   0.84,
  "similarity":   0.96,
  "hamming":      null,
  "app_name":     "네이버 지도"

```

| 필드 | 설명 |
|------|------|
| `track` | `"hash"` / `"fast"` / `"deep"` / `"error"` |
| `description` | 노년층 친화적 2문장 이내 설명 |

| `confidence` | OmniParser 탐지 신뢰도 |
| `similarity` | Fast Track 코사인 유사도 (hash/deep이면 null) |
| `hamming` | pHash 해밍 거리 (hash 트랙이면 0~8, 나머지 null) |

## 4. ChromaDB 사전 구축 — seed_db (Step 6-1)

### 배경

ChromaDB는 처음에 비어 있다. 즉, 첫 사용자가 아이콘을 탭하면 무조건 Deep Track(Claude Vision API 호출)이 실행된다. 이 문제를 해결하려면 **서비스 오픈 전에 주요 앱의 UI 요소를 미리 저장**해두어야 한다. 그래야 첫 사용자도 Fast Track 응답을 경험할 수 있다.

### seed_db.py 구조

각 UI 요소는 `UIElement` NamedTuple로 정의된다.

```python
class UIElement(NamedTuple):
    element_id: str      # 고유 식별자 (예: "baemin_nav_home")
    app: str             # 앱 이름 (예: "배달의민족")
    bbox: tuple          # (left, top, right, bottom) — 1080×2340px 기준
    element_type: str    # "icon" | "button" | "tab" | "text"
    label: str           # UI 요소 이름 (예: "홈 탭")
    description: str     # 노년층 친화적 목적 중심 설명
```

### 사전 구축 대상 — 3개 앱, 71개 요소

| 앱 | 패키지명 | 요소 수 | 주요 항목 |
|----|---------|---------|-----------|
| 배달의민족 | `com.baemin.android` | 33개 | 헤더 아이콘 4종, 검색창, 탭 5종, 음식카테고리 10종, 편의점 5종, 하단 탭바 5종, 팝업 닫기 |
| 네이버지도 | `com.nhn.android.nmap` | 17개 | 검색창, 길찾기 버튼, 카테고리 칩 4종, 지도 버튼 4종, 하단 탭바 5종 |
| 코레일 | `mobi.korail.Talk` | 21개 | 헤더 아이콘 3종, 승차권 입력 전체(교환·출발·도착·날짜·인원·간편구매·조회), 서비스 아이콘 4종, 하단 탭바 4종 |

### 시딩 파이프라인

```
스크린샷 PNG 로드
    ↓
각 UIElement의 bbox로 크롭 (scale_bbox로 실제 해상도에 맞게 자동 변환)
    ↓
CLIP embed_image() → 512차원 벡터
    ↓
phash_compute() → 64비트 pHash 계산
    ↓
ChromaDB 저장 (id, embedding, metadata{ app_package, description, phash, ... })
```

### bbox 자동 스케일링

모든 bbox 좌표는 **1080×2340px 기준**으로 정의되어 있다. 실제 스크린샷이 다른 해상도인 경우(예: 테스트에 사용된 648×1404px) `scale_bbox()`가 자동으로 비율 변환한다.

```python
def scale_bbox(bbox, src_w, src_h, ref_w=1080, ref_h=2340):
    l, t, r, b = bbox
    sx = src_w / ref_w   # 예: 648/1080 = 0.6
    sy = src_h / ref_h   # 예: 1404/2340 = 0.6
    return (int(l*sx), int(t*sy), int(r*sx), int(b*sy))
```

### 스크린샷 파일 처리

APP_CONFIG에는 `.png` 파일명으로 정의되어 있지만, 실제 파일이 `.jpg`인 경우에도 자동으로 폴백한다.

```python
if not screenshot_path.exists():
    alt = screenshot_path.with_suffix(".jpg")
    if alt.exists():
```

## 6. 3단계 캐시 파이프라인 — pHash Stage 추가

### 배경

CLIP 유사도 검색(Stage 2)은 완전히 동일한 이미지도 ~80ms가 소요된다. CLIP 임베딩 자체가 ~70ms, 벡터 검색이 ~10ms인 구조이기 때문이다. 그런데 사용자가 배달의민족을 쓰다가 같은 홈 탭을 또 탭하는 경우처럼, **완전히 동일하거나 거의 동일한 이미지**가 반복적으로 들어오는 케이스가 많다.

pHash(Perceptual Hash)를 첫 번째 단계로 추가하여 이런 케이스를 <1ms에 처리한다.

### pHash란

이미지를 64×64 그레이스케일로 축소한 뒤, DCT(이산 코사인 변환)를 적용하여 64비트 정수로 압축하는 방법이다. 두 이미지의 pHash 간 **해밍 거리(다른 비트의 수)**가 8 이하이면 "동일한 UI 요소"로 판정한다.

```
이미지 → 64×64 그레이스케일 → pHash(64비트) → 다른 이미지와 해밍 거리 비교
해밍 거리 0 = 완전히 동일 / 해밍 거리 8 이하 = 거의 동일 / 이상 = 다름
```

### hash_track.py 구조 (신규)

```python
HAMMING_THRESHOLD = 8

# 인메모리 스토어: element_id → (phash, description, element_type, app_package)
_store: dict[str, tuple] = {}
```

세 가지 핵심 함수:

**1) `search()` — Stage 1 캐시 검색**
```python
def search(image_bytes, app_package=""):
    ph = compute(image_bytes)           # 쿼리 이미지 pHash 계산
    for stored_ph, desc, el_type, pkg in _store.values():
        if app_package and pkg != app_package:
            continue                    # 다른 앱 제외
        dist = ph - stored_ph           # 해밍 거리
        if dist <= 8:
            return { "description": desc, "hamming": dist, ... }
    return None                         # 캐시 미스
```

**2) `load_from_collection()` — 서버 시작 시 일괄 로드**
```python
def load_from_collection(collection) -> int:
    results = collection.get(include=["metadatas"])
    for elem_id, meta in zip(ids, metas):
        phash_str = meta.get("phash", "")
        if phash_str:
            register(elem_id, from_str(phash_str), ...)
    return count  # 71개 로드
```

**3) `register()` — 실시간 등록 (Deep Track 저장 시)**
```python
def register(element_id, ph, description, element_type, app_package):
    _store[element_id] = (ph, description, element_type, app_package)
```

### 완성된 3단계 파이프라인

```
사용자 탭 → 크롭 이미지 + app_package 수신
    ↓
[OmniParser] element_type 탐지 (confidence)
    ↓
┌─ STAGE 1: pHash 해밍 거리 ≤ 8?  ──────────────────────────────────
│   YES → track:"hash", 즉시 반환 (<1ms)                              │
│   NO  ↓                                                              │
├─ STAGE 2: CLIP 코사인 유사도 ≥ 0.90 (+ app_package 필터)?  ─────  │
│   YES → track:"fast", 반환 (~80ms)                                  │
│   NO  ↓                                                              │
└─ STAGE 3: Claude Vision 설명 생성  ─────────────────────────────   │
    → ChromaDB 저장 (phash 포함)                                      │
    → pHash 인메모리 스토어 즉시 등록  ──────────────────────────────┘
    → track:"deep", 반환 (1~3s)
```

**중요**: Stage 3 저장 후 인메모리 스토어에 즉시 등록되므로, 같은 이미지가 다시 들어오면 서버 재시작 없이도 Stage 1에서 히트된다.

### seed_db와의 연동

`seed_db.py`가 pHash를 계산해서 ChromaDB 메타데이터에 저장하기 때문에, 서버 시작 시 `load_from_collection()`이 71개 pHash를 전부 스토어에 로드한다. **seed된 아이콘은 첫 번째 요청부터 Stage 1 히트**가 가능하다.

### 각 Stage 응답 속도 비교

| Stage | 방법 | 응답 시간 | API 호출 |
|-------|------|-----------|----------|
| Stage 1 (pHash) | 해밍 거리 비교 | < 1ms | 없음 |
| Stage 2 (CLIP) | 벡터 유사도 검색 | ~80ms | 없음 |
| Stage 3 (Deep) | Claude Vision | 1~3s | Claude API |

| `backend/pipeline/hash_track.py` | 신규 — pHash 인메모리 스토어 전체 구현 |
| `backend/main.py` | lifespan에 `phash_load()` 추가, 3단계 파이프라인으로 /capture 재구성, `hamming` 필드 응답 추가 |
